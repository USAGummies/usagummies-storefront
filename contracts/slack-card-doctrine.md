## 1. Why this doc exists

By 2026-05-02 the codebase has 7+ Slack cards (sales / email / finance
/ marketing / shipping / proposals / what-needs-ben / agents-status).
Without a shared doctrine they drift — one renderer uses `*Total*\nN`
fields, another forgets the read-only context note, a third invents a
new posture chip emoji. That drift is silent UX rot.

This doc + the `slack-card-builder.ts` primitives lock the standard so
**every future card lands compliant by default.**

---

## 2. The canonical card shape

Every Slack status card SHALL be assembled in this order:

```
┌─────────────────────────────────────────────────────┐
│ 1. Header        — emoji + title + posture chip      │
│ 2. Stats         — 6-field section (2-column grid)   │
│ 3. Brief         — "what this means" (1-2 lines)     │
│ 4. List sections — top rows / blockers (≤ 4 total)   │
│    (with dividers between them)                      │
│ 5. Context       — generated time + read-only note   │
│                    + degraded sources (if any)       │
│ 6. Actions       — link buttons only                 │
└─────────────────────────────────────────────────────┘
```

Cards may **omit** sections 4 + 6 when not applicable. They MUST NOT
reorder or duplicate sections.

---

## 3. Posture chips (HARD RULE)

Every card opens with one of four posture chips. Use the canonical
emoji/label exactly:

| Posture | Chip | Meaning |
|---|---|---|
| `green` | 🟢 clean | nothing waiting; no blockers |
| `yellow` | 🟡 work waiting | routine items; handle when ready |
| `red` | 🔴 attention | items at risk; operator action needed now |
| `unknown` | ⚪️ partial | some sources unavailable; can't claim green |

**Conservative-posture rule:** when a fetch fails, the lane shows
`unknown`, never `green`. Never claim "clean" on degraded data.

The shared helper `postureLabel()` in `slack-card-builder.ts` is the
single source of truth. Hand-rolled emojis are a doctrine violation.

---

## 4. Stats fields (HARD RULE)

Stats render as a 6-field section block. Each field uses the format
`*Label*\nValue`. Up to 10 fields (Slack truncates beyond ~10) but
**6 is the canonical count** — any more crowds the mobile view.

Numbers SHOULD have trailing-zero precision when meaningful
(`$42.00` not `$42`). Currency uses `$` not `USD`. Counts are bare
integers.

Helper: `statsBlock([{ label, value }, ...])`.

---

## 5. Brief block (HARD RULE)

The "what this means" sentence. **At most 1-2 lines.** Long-form
content lives in the dashboard, not in Slack.

The brief MUST adapt to posture:

- `green` → empty-state copy ("clean across all lanes")
- `yellow` → routine-action copy ("3 pending — review when ready")
- `red` → urgent copy with 🚨 emoji ("STALE approval (≥3d). Decide
  or stand it down.")

Helper: `briefBlock(text)`.

---

## 6. List sections (SOFT — at most 4)

Top rows / blockers / oldest pending — render as `listSectionBlock({
title, rows })` with each row prefixed `• `. The composer caps at 4
sections; cards needing more should link out to the dashboard
instead.

Strings inside list rows MUST truncate to 80 chars max via
`truncate(s)` helper. Slack mobile renders ≥80-char strings as
multi-line walls of text.

---

## 7. Context block (HARD RULE)

The context block is the ONLY place the following appear:

- `Generated HH:MMZ` — short ISO time, generated by `formatShortTime()`
- The **read-only assertion** — a one-line claim about what the card
  cannot do. Examples:
  - `Read-only — no QBO write fires from this card`
  - `Read-only — no label is bought from this card`
  - `Read-only — no execution fires from this card`
  - `Read-only — no external tool executes directly`
- Degraded sources (if any) — `:warning: Degraded: source: error · …`

Helper: `contextBlock({ generatedAt, readOnlyNote, degraded })`.

---

## 8. Actions block (HARD RULE — read-only)

Link buttons ONLY. Status cards MUST NOT include destructive action
buttons (approve / reject / send / publish). Those belong on
**dedicated approval cards** posted to `#ops-approvals` by
`requestApproval()`.

Each link's URL MUST be a canonical operator surface
(`/ops/<dept>/...` or `/api/ops/<surface>`).

Helper: `actionsBlock([{ text, url, actionId }, ...])`.

---

## 9. Banned anti-patterns

- ❌ Raw JSON dumps in `text:` block
- ❌ Email body excerpts > 12 lines (link to thread instead)
- ❌ Channel **names** (e.g. `#ops-daily`) in `postMessage({ channel })`
  — use canonical channel **IDs** via `slackChannelRef("ops-daily")`
- ❌ Top-level duplicate posts when a thread already exists — always
  reply in-thread for status updates
- ❌ Hand-rolled posture emojis or chip labels — always use the
  shared helpers
- ❌ Mutating buttons on status cards (see §8)

---

## 10. Migration plan

The cards shipped 2026-05-02 (`email-queue`, `finance-today`,
`marketing-today`, `shipping-today`, `external-proposals`,
`what-needs-ben`, `agents-status`) all match this shape **structurally**
— but they hand-roll the Block Kit JSON. Future cards SHALL use
`buildPostureCard()` from `slack-card-builder.ts` so the doctrine is
mechanically enforced.

Existing cards may be migrated opportunistically when their renderers
need other changes, but a wholesale refactor is **not** required —
the doctrine is enforced going forward, not retroactively.
